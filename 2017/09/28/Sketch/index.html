Sketch 上最好用的文件版本管理工具 — Abstract

题外话—为什么 Sketch 需要文件版本管理工具

很多时候的确是不需要文件版本管理，Sketch 本身就可以新增多个 Page 和 Artboard ，在设计项目较小的情况下，你完全可以把不同的版本放在不同的 Page 中，就实现了灵活的版本管理。但是当面临大型项目，单个版本无法完全放在一个 Page 中时，你就需要通过不同的 Sketch 文件来存放不同的版本，这时一个专门的版本管理工具就非常有必要了。

告别自己的文件目录

首先使用 Abstract 将短暂告别你自己的文件目录了，也就是说从新建Sketch 文件到修改不同的版本你都需要通过 Abstract 来打开你的 Sketch ，而Sketch 文件都保存在 Abstract 自己的私有目录中，然后再同步到 Abstract 自己的云端。

这样的好处是，对于大多数设计师的文件管理能力来说， Abstarct 无疑提供了更友好的方案来检索和管理你的设计资产。

坏处很明显，这让人感觉十分的不安全，不过好在你可以通过导出功能，把 Abstract 里的文件导出到你希望的文件夹里。

Abstract 的版本管理

先打开 Abstract 创建一个项目，创建完成后会需要选择是用 Abstract 新建一个 Sketch 文件，还是导入已有的文件。

接下来就需要引入一个重要的概念 「分支」了，可能有些设计师知道 Git ，这是工程师们拿来管理自己代码项目的工具，「分支」的概念就源于此。和我们平时做设计时的道理一样，在做大的修改前，我们一定会把原来的设计方案拷贝出一份来修改，这个拷贝出来的一份就是「分支」，而原来的不变那份就可以理解为「主干」。

回到软件中， Abstract 会默认给你建立一个 Master 主干（在一开始 Master 就是空的），这个主干你是不能直接修改的，你需要通过 Master 先建立一个分支，在分支中开始你的设计。通常我会先建立一个叫 init 的分支。

在分支中点击 「Edit in Sketch」Abstract 会帮你打开之前新建的 Sketch 文件，这时你可以在 Sketch 底部看到 Abstract 的插件已经开始跟踪你的设计稿，你每次 CMD + S 保存后的修改都会暂存起来。

当你修改得差不多以后，你就可以点击插件上的 Preview & Commit 按钮来确认之前的修改。Abstract 会在这时告诉你有哪些画板被修改或新增，而你需要填写一些描述来确认这次的修改，方便以后回滚的时候找到想要回滚到的版本。确认修改后 Abstract 会把你这次的修改同步到云端，同时在 Commits 中显示。

在一次次的 Commit 之后，你终于完成了主要的设计，虽然以后肯定也需要修改，但你已经确定它可以从「分支」变成「主干」了，这时点击分支中的 Merge Branch 就可以将分支合并到主干了，以后新的修改就从主干中再建立一个新的分支来修改。（值得说明的是，即便分支被合并掉了，你之前分支里的每一次 Commit 都是可以回滚恢复的。）

用 Abstract 来协作

Abstract 是支持多人团队协作的，其中最常用的一种协作模式就是为前端工程师提供最新的设计稿来切图。目前很多公司的前端工程师都是直接使用 Sketch 打开设计稿来进行切图，这时最烦人的问题就是开发途中会经常遇到产品需求变化或者工程技术问题，涉及到需要修改设计稿，修改完成后还要重新发给工程师，时间一长工程师的电脑里可能就已经保存着好几个版本的设计文件了。而使用 Abstract 云端同步功能，工程师能实时的打开最新的设计稿，并且可以不同步工程师在这个设计稿上的修改，并且通过设计师在 Commit change 中的描述和追踪到的有修改的画板，了解设计师修改了哪些部分。

另一个有意思的协作模式是作为设计师团队的共享资源库，将专门制作的设计资源文件放到 Abstract 中，方便设计师们协同更新和使用，每个人都可以参与设计资源的更新，更新后都可以通知到全员，而每个人使用的也都是最新版本的设计资源。另一方面即便不设置专门的共享资源库，设计师也能通过 Abstract 找到其他人的设计项目，灵活的从中找到自己需要复用的组件。

最后

其实了解 Coding 的设计师应该能看出来了，Abstract 是想做设计师的 Github。其实两者之间的差距还是蛮大的，别的不说，首先只能追踪到画板这一层的变化，可能就远远不够。另外 Github 除了用来管理代码，还是工程师之间源代码交流分享的地方，我也特别期待设计师们能有这么一个地方，大家不只是像在 Dribbble 那样通过一张只有 800x600 像素的图片来交流，我们还可以交流更完整的设计项目。
